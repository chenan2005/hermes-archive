# 3X-UI Reality Debugging Transcript

Real deployment: Alibaba Cloud Seoul (Ubuntu 24.04, 1.6GB RAM) + OpenWrt PassWall 4.67

## Environment

- **Server**: Alibaba Cloud ECS, Seoul, Ubuntu 24.04, IP 43.108.41.245
- **Open ports**: 40000-41000 (Alibaba security group)
- **3X-UI**: v3.3.1, Xray 26.6.1
- **OpenWrt**: PassWall 4.67-1, Xray 26.5.3

## Problem sequence and fixes

### 1. Initial deployment: config.json regeneration

After inserting inbound via SQLite and running `x-ui restart-xray`, the generated config.json had:
- `clients: null` — because the `clients` and `client_inbounds` tables were empty
- No `realitySettings.settings` — 3X-UI strips this field

Manual edits to config.json were wiped on every USR1 reload.

**Fix**: Populate all three tables (`clients`, `client_inbounds`, `client_traffics`), then `x-ui restart`.

### 2. Client shortId: "empty password" error

PassWall UCI options were wrong:
- Used `reality_pbk` instead of `reality_publicKey`
- Used `reality_sid` instead of `reality_shortId`
- Used `reality_spx` instead of `reality_spiderX`

The generated client config had `"shortId": ""`, causing Xray to reject with `empty "password"`.

**Fix**: Use correct UCI option names (verified against `util_xray.lua` line 97-146).

### 3. ShortId mismatch: "received real certificate"

Client had `shortId: a1b2c3d4` but server had `shortIds: [""]`. Server rejected the Reality handshake and fell through to dest, returning microsoft.com's real certificate.

**Fix**: Set server `shortIds: ["a1b2c3d4"]` to match client.

### 4. Flow mismatch: "account is not able to use the flow"

PassWall had `tlsflow=xtls-rprx-vision` on the node, but the server's client config (generated by 3X-UI) didn't include `flow`. Server error:
```
proxy/vless/inbound: account a5fa1889-... is not able to use the flow xtls-rprx-vision
```

**Fix**: Removed `tlsflow` from PassWall node config (`uci delete passwall.seoul_reality.tlsflow`). VLESS+Reality works without explicit flow — the Vision flow is server-side only.

### 5. PassWall restart kills SSH

Every `passwall restart` disrupted the SSH connection because PassWall modifies iptables rules (redirect/tproxy) during restart, which can briefly break outgoing connections including the SSH session.

**Workaround**: Use separate stop/start with waits, or accept the disruption and reconnect.

### 6. Multiple xray instances

Repeated `passwall start` without proper cleanup accumulated xray instances (4 at peak), causing port binding conflicts.

**Fix**: `killall -9 xray` before restart.

## Key takeaways

1. 3X-UI config generation is database-driven — always update SQLite tables, not config.json
2. PassWall Reality UCI options have specific names — verify against `util_xray.lua`
3. ShortId MUST be non-empty and MUST match between client and server
4. `tlsflow` is NOT needed for basic VLESS+Reality functionality
5. `realitySettings.settings` is NOT required on the server side
6. PassWall restart is disruptive to SSH — plan for reconnection
